Continuously Linting Ansible Playbooks and Roles

Fabian Köhler
2021-07-12 22:04
Programming
Ansible, CI, GitHub, Workflow, Action, YAML, Lint, Dependabot

ansible-lint is a helpful tool when dealing with Ansible roles and playbooks. It can help find common mistakes, prevent bugs and generally improve the code quality. Using the pre-commit framework it is trivial to set it up as hook in the Git version control system such that it checks the code before creating a commit. Sometimes however, one might forget to run pre-commit install after checking out a project or a collaborator is not using pre-commit. Therefore, it is helpful to set ansible-lint up as a GitHub workflow that is executed automatically on every commit and pull request. Of course this implies that one uses the GitHub platform. Other hosting services might offer other ways of achieving the same goal as well, but we focus on GitHub here.

In this repository an unofficial GitHub action can be found. However, I had no success with it. I identified the following problems:

  • It uses outdated versions of the ansible and the ansible-lint Python packages.
  • While the versions can be overridden I still could not get it to run.
  • The action creates a Docker container which makes the action rather slow.

Since ansible-lint is a simple Python package and creating a Docker image seems a bit like overshooting, I set up a GitHub workflow myself. I created the file .github/workflows/ansible-lint.yml (can be named differently as well) with the following content:

1
name: lint
2
on: [push, pull_request]
3
4
jobs:
5
  build:
6
    runs-on: ubuntu-latest
7
    steps:
8
      - uses: actions/checkout@v2
9
      - name: Set up Python 3.9
10
        uses: actions/setup-python@v2
11
        with:
12
          python-version: 3.9
13
      - uses: actions/cache@v2
14
        with:
15
          path: ~/.cache/pip
16
          key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements.txt') }}
17
      - name: Install dependencies
18
        run: |
19
          python -m pip install --upgrade pip
20
          python -m pip install -r requirements.txt
21
      - name: Run ansible-lint
22
        run: |
23
          ansible-lint -v --show-relpath
This instructs the workflow to run on every pushed commit and every pull request. First the action actions/check@v2 is used to set up the environment and checkout the repository. The action actions/setup-python@v2 sets up Python, version 3.9 in my case. Then I use the actions/cache@v2 to cache the Python dependencies I install in the following step. I have a requirements.txt file in the root of the repository to track the required Python packages:
1
    ansible==4.2.0
2
    ansible-lint==5.0.12
The line path: ~/.cache/pip instructs the CI to cache the pip directory and key: … determines a key for the cache which is based on the operating system as well as the hash of the requirements.txt. As long as the OS and the file do not change, the downloaded dependencies can be reused in the next step Install dependencies thus speeding up the workflow. Of course one could also manually install ansible-lint with python -m pip install ansible-lint but that would require figuring out a different caching mechanism and would prevent Dependabot from checking for updates (see below).
In the last step Run ansible-lint the linter is actually executed. Here, one could also change the CLI arguments used when calling ansible-lint, however it is much simpler to use a .ansible-lint file in the repository to configure it. More on this topic can be found in the official documentation. For example, it might be useful to exclude certain files that are not actually Ansible code:
1
    exclude_paths:
2
      - ".pre-commit-config.yaml"
3
      - "roles/vps/files/docker-compose-files"
As you can see, setting up ansible-lint as a GitHub action is pretty straightforward. I have two final recommendations that you might find useful. Dependabot can be used to check the packages used in the requirements.txt file for updates. Just create a file .github/dependabot.yml with the content
1
    version: 2
2
    updates:
3
      - package-ecosystem: "pip"
4
        directory: "/"
5
        schedule:
6
          interval: "daily"
to enable a daily update check. If Dependabot finds an update it will create a pull request for the repository. You can add a nice badge to the README.md file indicating the workflow status using Shields.io. Select the "GitHub Workflow Status" and specify your repository and the workflow name. The name is specified by the first line in the workflow file (name: lint) so in my case it would be lint. Shields.io will then generate links, markdown, etc. to easily use the badge.
